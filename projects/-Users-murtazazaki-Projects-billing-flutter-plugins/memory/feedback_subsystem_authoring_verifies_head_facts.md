When authoring code that depends on subsystem facts (table existence, RPC signatures, KV cache key prefixes, filter predicates, vault auth patterns, skill-mirror locations), grep / read / SQL HEAD evidence first. CLAUDE.md prose and plan text describe intent; only HEAD describes reality.

**Why:** On 2026-05-03 the user rejected an `ExitPlanMode` call for a tutorials banner-pool plan because the plan made six independent HEAD-fact mistakes that would have caused production incidents:

1. **Audit-table assumed to exist.** Plan called `INSERT INTO admin_tutorial_audit_log (...)` but grep found only `banner_audit_log` and `payment_audit_log` in HEAD. The `admin_tutorial_audit_log` table is referenced in `supabase/migrations/CLAUDE.md:86` as a future requirement only. The migration would have crashed at apply time.
2. **KV cache key prefix stale.** Plan purged `banner:v3:*`. Actual Worker key is `banner:v3.1:*` (`microservices/tutorials/src/utils/cache-keys.ts:39`, bumped 2026-04-23 in PR-β SCHEMA-LOCK-357). Production purge would have missed all current keys; stale broken-URL responses would have served for up to 30 minutes.
3. **RPC parameter count wrong.** Plan claimed `search_reels_v1` has 8 params. Actual signature has 6 (`p_query, p_cursor, p_limit, p_build_number, p_signed_in, p_surface_id`).
4. **RPC canonical body location wrong.** Plan based the forward-fix on `000441`. Actual canonical body is in `000443_search_reels_v1_cursor_hardening.sql` which adds the `_search_reels_safe_cursor` helper. Forward fix would have lost the helper integration.
5. **Filter predicate misread.** Plan asserted `get_banner_recommendation_v5` is "fully permissive". Actual WHERE clause at `000442:1127-1133` includes `AND t.is_media_ready = TRUE` (added by 000442). The pool filters `is_media_ready=false` already; only `is_reels_eligible=false` passes through. Doctrine non-regression test would have been over-broad.
6. **Skill-mirror dual root.** Plan only edited `.claude/skills/tutorials-manager/data/rules/corpus.csv`. The `.agents/skills/tutorials-manager/data/rules/` parallel root exists (Codex CLI / OpenCode side) and ships its own corpus.csv. Edits must mirror or be symlinked.

Plus a Vault auth pattern correction: pg_net trigger calls in HEAD (`000297`, `000301`, `000436`) use `public.get_secret(...)` from the vault helpers, not `current_setting(...)`. Migrations following the `current_setting` pattern would silently fall through on missing GUCs.

**How to apply:**
- For any new audit-log envelope row: grep `CREATE TABLE.*<table_name>` in `supabase/migrations/*.sql` first. If absent, the migration must CREATE the table at the top.
- For any KV purge call: read `microservices/tutorials/src/utils/cache-keys.ts` (or the equivalent for other Workers) for the live key shape, never trust CLAUDE.md cache-TTL tables which can be one bump behind.
- For any RPC `CREATE OR REPLACE FUNCTION`: read the most recent forward-fix migration (search `grep -l "<rpc_name>" supabase/migrations/*.sql | sort -V | tail -3`) for the canonical body, parameter count, and proconfig (`SET search_path`, `SET statement_timeout`). `CREATE OR REPLACE` does NOT preserve `SET` clauses across redefinition.
- For any filter-predicate claim: open the migration at the cited line range and read the WHERE clause verbatim. Filter predicates can be tightened by intermediate forward fixes (e.g., 000442 added `is_media_ready=TRUE` to v5).
- For any skill-rule edit: `ls -la .claude/skills/<skill>/data/rules/` AND `ls -la .agents/skills/<skill>/data/rules/`. If both exist and are not symlinked at the directory level, edit both.
- For any pg_net call in a migration: grep for `public.get_secret` calls in recent migrations (000297, 000301, 000436 are exemplars) — use the Vault helper pattern, not `current_setting(...)`.
- For any Vault best-effort pg_net plan: BEFORE relying on it as the primary KV invalidation path, verify the GUC keys are populated against linked prod via `mcp__claude_ai_Supabase__execute_sql` with `SELECT public.get_secret('<key_name>')`. If NULL, plan `make admin-purge-banner-all ENV=production` (with `CACHE_PURGE_TOKEN` exported from 1Password / wrangler secret list) as the PRIMARY purge path, not the fallback. Confirmed 2026-05-03: tutorials Worker's `tutorials_worker_admin_purge_url` + `cache_purge_token` are NOT set in linked prod, so 000445's inline pg_net silently no-op'd; operator's shell did NOT have `CACHE_PURGE_TOKEN` exported either, so the manual fallback also failed. Worker KV held stale `/placeholder.webm` responses until natural TTL expiry (30m banner, 60s search, 1h playlist, 10m by-id).
- **Vault secret namespace convention** (codified 2026-05-03 after `list_secrets() = []` discovery): when authoring a new pg_net auto-purge migration for the tutorials Worker, REUSE the established secret pair `tutorials_worker_url` (BASE URL, no trailing slash, no path) + `cache_purge_token`, and append `/v1/admin/purge?pattern=<X>` at the call site (pattern in querystring, empty body). This is the convention used by 000409 baseline, 000411 banner-purge trigger, 000436 GST sentinel trigger. Do NOT introduce new secret names like `tutorials_worker_admin_purge_url` (the 000445 deviation that fragmented the namespace and forced multiple Vault populations) — pattern in JSON body is fine but the URL secret should remain the BASE URL.
- **Vault writes cannot be performed via Supabase MCP** (confirmed 2026-05-03): even though the MCP role is `postgres` and has EXECUTE on `public.upsert_secret`, the underlying `vault.secrets` INSERT trigger calls `pgsodium._crypto_aead_det_noncegen` which is ACL'd to `pgsodium_keyholder` only. SECURITY DEFINER does not escalate past pgsodium grants. Operator must run Vault `upsert_secret` calls in Supabase Studio SQL Editor (which uses `supabase_admin` with proper pgsodium grants) or via direct `psql` with elevated credentials. MCP is fine for Vault READS (`SELECT public.get_secret(...)`, `SELECT * FROM public.list_secrets()`).
